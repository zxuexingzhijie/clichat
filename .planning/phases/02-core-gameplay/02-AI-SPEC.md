## 1. System Classification

**System Type:** Content Generation + Structured Retrieval (Hybrid)

**Description:**
AI驱动的游戏内容生成系统，为CLI文字RPG提供三种AI角色：Narrative Director（场景叙述）、NPC Actor（角色对话）、Retrieval Planner（上下文检索规划）。玩家通过终端与世界交互，AI根据Rules Engine的裁决结果生成80-180字中文叙述和NPC对话。"好"的输出 = 叙述风格适配场景、NPC人格一致、严格基于Codex事实、检定结果与叙事衔接。

**Critical Failure Modes:**
1. **世界事实捏造** — AI编造Codex中不存在的NPC、地点、魔法或历史事件，破坏世界一致性
2. **游戏状态覆写** — AI叙述暗示机械效果（获得物品、HP变化、关系变化）但Rules Engine未授权
3. **NPC人格坍缩** — 同一NPC在不同场景中性格不一致，或泄露不应知道的信息
4. **检定结果矛盾** — 叙事描写与Rules Engine判定的成功/失败结果相矛盾
5. **Prompt注入突破** — 玩家通过NL输入操控AI生成违规内容或绕过系统约束

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** 游戏 — AI驱动的交互式叙事 / 文字RPG

**User Population:** 中文文字RPG玩家。通过CLI终端与游戏世界交互，期望沉浸式叙事体验，而非聊天机器人式的问答。玩家对叙事质量、角色一致性和世界可信度有较高要求。

**Stakes Level:** Medium

**Output Consequence:** AI输出直接构成玩家的游戏体验——场景叙述、NPC对话、战斗描写。低质量输出不会造成法律或安全风险，但会直接破坏沉浸感，导致玩家流失。AI输出被玩家即时消费，无人工审核环节；错误在玩家面前实时暴露。

### What Domain Experts Evaluate Against

```
维度: 世界事实忠实度
好（专家认可）: 叙述严格基于Codex中已定义的世界设定——地名、种族特征、魔法规则、
  NPC关系均与既定事实一致。例如"铁匠提到了黑松镇的矿石短缺"，而矿石短缺确实
  记录在地区设定中。
坏（专家标记）: AI凭空发明世界事实——编造不存在的NPC、创造未定义的魔法、
  引用虚构的历史事件。例如叙述"你想起了上次与精灵女王的会面"，但Codex中
  该地区根本没有精灵女王。
重要性: Critical
来源: AI Dungeon生产经验——"世界遗忘"是玩家投诉排名第一的问题。
  AI生成的叙述如果与已建立的世界设定矛盾，会瞬间打破沉浸感。
  设计原则"Codex管真相"要求AI绝不发明世界事实。
```

```
维度: NPC人格一致性
好（专家认可）: 同一NPC在不同场景中保持一致的说话风格、性格特征、知识边界和
  动机。例如"沉默寡言的老兵"始终用短句、回避过去、不主动闲聊——即使多轮对话
  后也不会突然变成话痨。NPC只谈论自己应该知道的事情。
坏（专家标记）: NPC人格漂移——同一角色在不同对话中性格不一致、突然掌握不应知道
  的信息、或在没有剧情驱动的情况下态度180度转变。例如刚拒绝帮助玩家的商人，
  下一句突然主动提供任务线索。
重要性: Critical
来源: Character.AI和NovelAI社区反馈——角色一致性是AI角色扮演系统中用户满意度
  最强的预测因子。Disco Elysium的设计经验表明，NPC可信度来自于行为约束而非
  信息丰富度。设计原则"NPC管偏见"要求每个NPC有自己的认知边界。
```

```
维度: 叙述风格适配
好（专家认可）: 叙述风格根据场景类型自动切换且切换自然——战斗场景用短句强动作
  （"长剑划过，鲜血飞溅"），探索场景有氛围感（"月光从破碎的穹顶洒下，照亮
  满地的碎石"），对话像轻小说式自然口语（"你这家伙，又来了？"）。风格切换
  不生硬，同一回合内保持调性统一。
坏（专家标记）: 风格错位——战斗场景用长篇抒情描写拖慢节奏，日常对话过于
  文绉绉像在背课文，探索场景干巴巴只有功能性描述。或者同一段文本内风格
  来回跳跃，前半句是武侠小说后半句是网络口语。
重要性: High
来源: 交互式叙事设计原则——文字RPG的"手感"主要来自叙述节奏。AI Dungeon和
  NovelAI的用户研究显示，风格不一致比偶尔的事实错误更影响玩家对"AI质量"
  的整体感知。项目决策D-07明确定义了五种场景调性。
```

```
维度: 中文语言质量
好（专家认可）: 80-180字中文叙述流畅自然，用词准确，句式有变化，不出现
  翻译腔或机器味。战斗用词有力度（"劈""斩""轰"），探索用词有画面感
  （"幽暗""蜿蜒""苔藓斑驳"），对话用词口语化且符合角色身份。
坏（专家标记）: 翻译腔——"你感到一种强烈的紧迫感"（英转中直译）；
  重复套路——每段叙述都以"你注意到"开头；机器味——用词过于书面化或堆砌
  形容词；字数失控——远超180字或不到40字。
重要性: High
来源: 中文AI内容生成的已知挑战——LLM的中文输出常带英语思维结构。
  中文网文/轻小说社区对"AI味"极为敏感，翻译腔和重复句式是最常被识别的
  AI生成标志。项目决策D-10明确了80-180字限制。
```

```
维度: 检定结果叙事衔接
好（专家认可）: 机械判定结果（[D20:14]+体魄3=17 vs DC15 → 成功）与
  叙事描写之间的衔接自然合理——成功程度反映在叙事强度上（勉强成功vs大成功
  的描写有明显区别），检定原因清楚（为什么需要掷骰），叙事不与判定结果矛盾。
坏（专家标记）: 叙事与检定脱节——判定显示"失败"但叙述描写了成功的动作；
  成功程度不匹配——大成功和普通成功的叙述完全相同；叙事忽略检定——判定后
  的叙述完全没有体现判定结果，仿佛检定没发生过。
重要性: Critical
来源: TTRPG（桌面角色扮演）设计经验——检定与叙事的一致性是"骰子管命运"
  原则的核心体现。当叙事与机械结果矛盾时，玩家会质疑整个系统的可信度。
  项目决策D-17明确了"检定先显示、叙述后呈现"的策略。
```

### Known Failure Modes in This Domain

**1. 世界遗忘（World Amnesia）**
AI在生成叙述时忽略或矛盾已建立的世界事实。最常见的表现：地名不一致（"黑松镇"变成"松林镇"）、NPC关系矛盾（刚和玩家交过手的NPC表现得像第一次见面）、空间逻辑错误（叙述中提到的地理方位与地图不符）。这是所有AI交互式叙事系统排名第一的用户投诉类型（AI Dungeon、NovelAI均有大量此类反馈）。

**2. NPC人格坍缩（Character Collapse）**
长时间对话后NPC逐渐失去独特性，所有NPC开始"说一样的话"。根本原因是上下文窗口中角色约束被稀释——越多对话历史进入窗口，NPC个性模板的影响力越弱。表现为：独特口头禅消失、不同NPC给出相似的建议和措辞、情绪状态不再影响对话风格。

**3. 游戏状态覆写（State Override）**
AI叙述暗示或明确声明了不属于它管辖的机械效果——例如在叙述中写"你获得了一把魔法剑"但实际游戏状态中没有这把剑，或叙述"敌人倒下了"但Rules Engine判定敌人仍有HP。这违反了"AI不决定事件成败"的架构边界。在AI Dungeon中这是导致"剧情Bug"的首要原因。

**4. 叙述节奏失控（Pacing Breakdown）**
战斗场景的叙述过长导致紧张感消失（每回合读200字战斗描写等于没有战斗节奏），或探索场景叙述过短导致世界感缺失（"你看到一间房间。里面有东西。"）。字数限制（80-180字）只是底线，真正的问题是信息密度和节奏感是否匹配场景类型。

### Regulatory / Compliance Context

无直接适用的行业监管要求。以下约束来自产品设计而非法规：

- **内容安全**: AI生成的中文叙述和NPC对话不得包含违反中国互联网内容法规的内容（暴力/色情过度渲染、政治敏感内容）。这不是游戏行业特有的，而是所有面向中国用户的AI生成内容的通用要求。
- **玩家输入注入防护**: 玩家可能通过自然语言输入尝试prompt注入，让AI输出不当内容或绕过Rules Engine。架构已通过"AI不决定机械结果"的边界防御这一点，但叙述内容本身仍需防护。
- **未成年人保护**: 如果游戏面向未成年玩家，AI生成内容需要符合相应分级标准。当前为单机CLI游戏，风险较低，但应作为内容安全策略的一部分考虑。

## 2. Framework Decision

**Selected Framework:** Vercel AI SDK v5

**Version:** ai@^5.0.179 (with @ai-sdk/openai, @ai-sdk/anthropic, @ai-sdk/google, @ai-sdk/alibaba, @ai-sdk/deepseek)

**Rationale:**
项目已使用 AI SDK v5 作为多提供商抽象层。本阶段的AI系统本质是受约束的内容生成（不是自主Agent），直接使用 `generateText`/`generateObject` + Zod schema 即可满足所有需求。零新依赖、TypeScript原生、多提供商路由（快模型做检索/过滤，好模型做叙述/对话）。

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| LangChain.js | CLAUDE.md 明确标注"Over-engineered for this use case"。无 vector DB = 检索优势为零。引入重依赖与已有 AI SDK 冲突。 |
| LangGraph | 游戏已有自己的状态管理(createStore)和循环(game loop)。图模型与 turn-based 游戏不匹配，增加复杂度不增加价值。 |
| OpenAI Agents SDK | 本项目需要多提供商(Qwen/DeepSeek做中文)，OpenAI vendor lock-in 不可接受。 |

**Vendor Lock-In Accepted:** No — AI SDK v5 是 provider-agnostic 的，可随时切换底层模型。

### Key Abstractions
| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `generateText` | 生成纯文本，支持 streaming | Narrative Director 叙述生成 |
| `generateObject` | 生成符合 Zod schema 的结构化 JSON | NPC对话、Retrieval Plan、Safety Filter |
| `streamText` | 流式文本输出 (AsyncIterable) | 叙述逐字展示给玩家 (UX) |
| `Output.object({ schema })` | v5 的结构化输出模式 | 替代旧版 `experimental_generateObject` |
| Provider functions | `google()`, `openai()` 等 | 按角色路由到不同模型 |

### Common Pitfalls
1. **AI SDK v5 vs v6 API 差异** — v6 废弃了 `generateObject`，改用 `generateText` + `Output.object`。本项目锁定 v5，使用 `generateObject` 直接调用。
2. **Zod v4 兼容性** — AI SDK v5 要求 zod `^3.25.76 || ^4.1.8`。项目用 zod 4.3.6，满足约束。
3. **`maxTokens` 必须显式设置** — 不设置则使用 provider 默认值（可能很大），浪费 token 预算。每个角色明确设定上限。
4. **Streaming 不支持结构化输出** — `streamText` 只能用于纯文本。NPC对话等需要结构化数据的场景必须用 `generateObject`。
5. **Provider 环境变量** — 每个 provider 需要对应的 API key 环境变量（OPENAI_API_KEY, GOOGLE_GENERATIVE_AI_KEY, ANTHROPIC_API_KEY 等）。

## 4. Implementation Guidance

**Model Configuration:**

| AI 角色 | 模型 | Temperature | Max Tokens | 延迟要求 |
|---------|------|-------------|------------|----------|
| Narrative Director | Gemini 2.0 Flash / Qwen-Plus | 0.7 | 512 | <2s 首token |
| NPC Actor | Gemini 2.0 Flash / Qwen-Plus | 0.8 | 400 | <2s 完成 |
| Retrieval Planner | Gemini 2.0 Flash | 0.1 | 200 | <500ms |
| Safety Filter | Gemini 2.0 Flash | 0.0 | 50 | <300ms |
| Summarizer (后台) | DeepSeek Chat | 0.3 | 800 | 无限制 |
| Quest Planner (后台) | Claude Sonnet | 0.6 | 2000 | 无限制 |

**Core Pattern:**

```typescript
// src/ai/providers.ts
type AiRole = 'narrative-director' | 'npc-actor' | 'retrieval-planner'
  | 'safety-filter' | 'summarizer' | 'quest-planner';

const MODEL_MAP: Record<AiRole, () => LanguageModel> = {
  'narrative-director': () => google('gemini-2.0-flash'),
  'npc-actor': () => google('gemini-2.0-flash'),
  'retrieval-planner': () => google('gemini-2.0-flash'),
  'safety-filter': () => google('gemini-2.0-flash'),
  'summarizer': () => deepseek('deepseek-chat'),
  'quest-planner': () => anthropic('claude-sonnet-4-6'),
};

export function getModel(role: AiRole): LanguageModel {
  return MODEL_MAP[role]();
}
```

**Tool Use:**
本阶段AI角色不使用 AI SDK 的 tool calling 功能。Retrieval Planner 输出结构化 JSON（指定需要检索的 codex ID/tag），由游戏代码执行实际文件查询。AI 不直接访问文件系统。

**State Management:**
AI 角色无状态——每次调用是独立的。上下文通过 Context Assembler 从游戏状态（stores）中提取并组装为 prompt。NPC 记忆存储在 JSON 文件中，由游戏代码管理，按需注入 prompt。

**Context Window Strategy:**
- 固定 token 预算分配：system(300) + codex(500) + npc_memory(150) + recent_narration(300) + player_action(50) + buffer(100) = ~1400 tokens input
- 叙述历史：滑动窗口 3 回合，场景切换时清空
- NPC 记忆：最多 3 条最相关记忆，超过 10 条时由 Summarizer 后台压缩
- Codex 条目：Retrieval Planner 限制每回合最多 3 个条目，每条截断至 200 字符

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns for this game AI system.

### 4b.1 Structured Outputs with Zod

AI SDK v5 使用 `generateObject` + Zod schema 保证输出结构。LLM 被指示以 JSON 输出 → AI SDK 解析 → Zod 验证 → 返回类型化对象。

```typescript
// src/ai/schemas/npc-dialogue.ts
import { z } from 'zod';

export const NpcDialogueSchema = z.object({
  dialogue: z.string().min(10).max(300).describe('NPC对白，自然口语'),
  emotionTag: z.enum(['neutral', 'happy', 'angry', 'sad', 'fearful', 'amused', 'suspicious']),
  shouldRemember: z.boolean().describe('是否将此次互动写入NPC长期记忆'),
  relationshipDelta: z.number().min(-0.5).max(0.5).describe('关系值变化建议'),
});

export type NpcDialogue = z.infer<typeof NpcDialogueSchema>;
```

**重试逻辑：** 最多重试 2 次（共 3 次调用）。重试时在 prompt 末尾追加上次的校验错误信息。超过重试次数后抛出错误，由调用层决定降级策略（如显示兜底叙述文本）。

### 4b.2 Async-First Design

**Stream vs. Await 选择：**

| 场景 | 选择 | 原因 |
|------|------|------|
| Narrative Director 叙述 | `streamText` | 玩家逐字看到叙述出现，UX更好 |
| NPC Actor 结构化对话 | `generateObject` | 需要完整 JSON，不能用 partial |
| Retrieval Planner | `generateObject` | 需要完整 tool call 结果 |
| Summarizer (后台) | `generateText` | 无 UX 需求，需要完整结果 |

**React/Ink 中的 async 模式：** 在 `useEffect` 中使用 `.then()` 链或 async IIFE，设置 `cancelled` flag 防止组件卸载后的状态更新。

### 4b.3 Prompt Engineering Discipline

**System vs. User prompt 分离：**
- `system`: AI 角色定义、约束规则、输出格式、风格指令（按 sceneType 切换）
- `prompt`: 场景信息、Codex 条目、NPC 记忆、玩家动作、检定结果

**Prompt 模板化：** 每种场景类型有独立的风格指令字符串，运行时按 sceneType 选择注入 system prompt。

**Token budget 显式设置：** 所有 `generateText`/`generateObject` 调用必须显式设置 `maxTokens`。

### 4b.4 Context Window Management

每回合上下文组装流程：
```
Retrieval Planner (决定取什么) → Codex Query (精确查询) → Context Assembler (组装+截断) → AI Role
```

- Codex 条目截断至 200 字符/条，最多 3 条
- NPC 记忆最多 3 条，超 10 条由 Summarizer 压缩
- 叙述历史固定 3 回合滑动窗口
- 场景切换时清空叙述历史

### 4b.5 Cost and Latency Budget

**单次 session 估算（~50 回合）：** ~$0.03-0.05，目标 < $0.10

**成本控制：**
1. Retrieval Planner + Safety Filter 用最便宜的 Flash 模型
2. 相同场景+意图的 Retrieval Plan 缓存 60 秒
3. System prompt 利用 provider 的 prompt caching（Anthropic/Google 原生支持）
4. Summarizer 后台异步，用最便宜模型，失败不影响游戏

**延迟目标：**
| 路径 | 目标 | 策略 |
|------|------|------|
| 玩家输入 → 叙述首token | <2s | streamText 流式 |
| 玩家输入 → NPC对话完成 | <2s | generateObject |
| 回合 → 检索计划 | <500ms | 轻量prompt + 缓存 |
| 内容安全过滤 | <300ms | 最小prompt + 二元判定 |

## 5. Evaluation Strategy

### Dimensions

| Dimension | Rubric (Pass/Fail or 1-5) | Measurement Approach | Priority |
|-----------|--------------------------|---------------------|----------|
| 世界事实忠实度 | Pass: 叙述中所有实体名、地点、关系均可在Codex中找到对应条目。Fail: 出现任何Codex中不存在的世界事实 | Code（Codex条目交叉检验）+ LLM Judge | Critical |
| NPC人格一致性 | 1-5: 5=完全符合NPC身份/口头禅/知识边界, 1=完全偏离角色设定 | LLM Judge（对比NPC profile与生成对话）| Critical |
| 检定结果叙事衔接 | Pass: 叙事结果方向与检定判定一致且强度匹配。Fail: 叙事与判定矛盾 | Code（解析检定结果+关键词匹配）+ Human | Critical |
| 游戏状态边界 | Pass: 叙述中不包含未授权的机械效果声明。Fail: 叙述暗示物品获得/HP变化/关系改变 | Code（正则检测机械效果关键词）| Critical |
| 叙述风格适配 | 1-5: 5=完美匹配场景类型的节奏和调性, 1=风格完全错位 | LLM Judge（场景类型+风格标准对比）| High |
| 中文语言质量 | 1-5: 5=流畅自然无翻译腔, 1=明显机器味/重复句式/字数超限 | LLM Judge + Human（定期采样）| High |
| 字数合规 | Pass: 80-180字。Fail: 超出范围 | Code（`string.length` 检查）| Medium |
| 响应延迟 | Pass: <2s首token(叙述), <2s完成(NPC)。Fail: 超时 | Code（计时器）| Medium |

### Eval Tooling

**Primary Tool:** Promptfoo (CLI-first, 适合 CI/CD) + 自定义 Code Validators

**Setup:**
```bash
bun add -d promptfoo
# promptfoo.yaml 配置见 tests/ai-eval/
```

**CI/CD Integration:**
```bash
# 在 bun test 之后运行 AI eval suite
bun run test:ai-eval
# 等价于: promptfoo eval --config tests/ai-eval/promptfoo.yaml
```

**选择 Promptfoo 而非 Arize Phoenix 的理由：** 本项目是 CLI 单机游戏，无生产环境服务端。Promptfoo 的 CLI-first 设计更匹配开发流程（本地运行 eval、CI/CD 集成）。Phoenix 更适合有在线服务的系统。

### Reference Dataset

**Size:** 20 examples to start (每种 AI 角色 + 场景类型组合)

**Composition:**
- 5x Narrative Director 探索场景（含 Codex 事实验证）
- 5x Narrative Director 战斗场景（含检定结果衔接验证）
- 5x NPC Actor 对话（不同NPC性格 + 情绪状态）
- 3x Retrieval Planner（不同场景复杂度）
- 2x 边缘案例（玩家尝试 prompt 注入、超出预期的输入）

**Labeling:**
游戏叙事设计师标注"好/坏"样本 + 具体原因。LLM Judge 校准后与人工标注对比，目标相关性 ≥ 0.7。

## 7. Production Monitoring

**Tracing Tool:** 自定义日志系统（文件级JSON日志，非在线平台）

本项目是单机CLI游戏，无服务端。"生产监控"指的是开发期间的质量追踪和玩家自愿反馈收集。

**Key Metrics to Track:**
1. AI 调用延迟（p50, p95）— 确保游戏节奏不被 API 延迟拖垮
2. Schema 验证失败率 — 追踪哪些模型/角色经常输出不合格JSON
3. 重试次数 — 如果某角色经常需要重试，说明 prompt 或 schema 需要优化
4. Token 使用量/session — 成本追踪，防止 prompt 膨胀
5. 兜底文本触发率 — AI 生成完全失败时的降级频率

**Alert Thresholds:**
- Schema 验证失败率 > 10% → 检查 prompt template
- 平均延迟 > 3s → 检查模型选择或网络问题
- 兜底触发率 > 5% → 紧急排查

**Smart Sampling Strategy:**
- 每个 session 的首 5 回合必须记录完整 AI 输入输出（冷启动质量）
- 战斗场景 100% 记录（检定衔接是最易出错的）
- NPC 对话超过 5 轮的 100% 记录（人格坍缩风险）
- 其他场景 20% 随机采样
