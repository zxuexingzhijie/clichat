- CodexEntrySchema: z.discriminatedUnion('type', [...all 8 schemas])
- Export all individual schemas and the union, plus inferred types via z.infer

Create `src/codex/schemas/relationship.ts` per D-31:

- `queryById(entries: Map<string, CodexEntry>, id: string)`: Lookup by id
- `queryRelationships(edges: RelationshipEdge[], filter: {source_id?: string, target_id?: string, relation_type?: string})`: Filter relationship edges by any combination of fields

| Boundary | Description |
|----------|-------------|

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-06 | Tampering | loader.ts | mitigate | All YAML data validated through Zod schema before use; unknown fields rejected via z.strict() or z.passthrough() with explicit filtering; no eval() or dynamic code from data files |
| T-01-07 | Tampering | YAML files | mitigate | YAML files are local game assets, not user-uploaded content. Zod validation catches any malformed entries. Error messages include file path and entry id for debugging. |
| T-01-08 | Information Disclosure | epistemic.ts | accept | Visibility field ('hidden', 'secret', 'forbidden') is a data tag for game logic, not a security control. In Phase 1, all data is local and single-player. Epistemic filtering for NPC cognition boundaries is Phase 2+. |
</threat_model>
