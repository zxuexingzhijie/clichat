# v1.2: Code Review Report

**Reviewed:** 2026-04-29T07:24:50Z
**Depth:** standard

## Summary

v1.2 is functionally coherent at a high level. The quest-system event wiring, memory retention, and save/restore pipeline are all correctly structured. Most findings below are integration gaps or silent failure paths that would confuse a player — specifically around the free-text dialogue path missing its `dialogue_ended` event, the scene-manager's hardcoded singleton store usage, and the quest-system `stage_expose` trigger logic which will never fire the completion in the normal play-through.

## Critical Issues

### CR-01: `stage_expose` quest completion is permanently blocked

**File:** `world-data/codex/quests.yaml:43-59` / `src/engine/quest-system.ts:160-170`

**Issue:** `stage_expose` (the final stage of `quest_main_01`) has `secondaryEvent: item_found` with no `secondaryTargetId`. The `checkAndAdvance` logic in `quest-system.ts` requires **both** `primary` (dialogue with `npc_elder`) and `secondary` (`item_found`) to have fired before it calls `completeQuest`. However, the item `item_evidence` is acquired during `handleInspect` in `scene-manager.ts`, which fires `item_acquired`, not `item_found`. The `bus.on('item_acquired', ...)` handler inside `quest-system.ts` maps `item_acquired` to the `item_found` trigger type — that part is correct. The problem is ordering: the player most likely picks up the evidence **before** reaching `stage_expose` (they are in `stage_investigate` at that point). When `item_acquired` fires during `stage_investigate`, the stage's trigger is `location_entered`, so the `item_found` arm is skipped entirely. By the time the quest advances to `stage_expose`, the `item_found` condition will never fire again (the item is already in the player's tags and won't re-trigger `item_acquired`). The `pendingConditions` map key is `questId:stageId`, so the secondary signal for `stage_expose` is lost permanently.

**Fix:** Either (a) assign `secondaryTargetId: item_evidence` to `stage_expose` **and** ensure the quest system fires a synthetic secondary-check when entering the stage if the item is already held, or (b) change `stage_expose` to use only `dialogue_ended` as its primary trigger (simplest and sufficient — the player can only talk to the elder after finding evidence in the cave anyway given the scene gating):

```yaml
# option b — simpler fix
- id: stage_expose
  description: 揭露狼灾真相
  trigger:
    event: dialogue_ended
    targetId: npc_elder
  # remove secondaryEvent entirely
  objectives:
    - id: obj_talk_elder
      ...
    - id: obj_find_evidence
      ...
  nextStageId: null
```

## Warnings

### WR-01: `scene-manager.ts` imports singleton stores instead of the injected ones

**File:** `src/engine/scene-manager.ts:4-5`, `src/engine/scene-manager.ts:112`, `src/engine/scene-manager.ts:141`, `src/engine/scene-manager.ts:193`, `src/engine/scene-manager.ts:271`

**Issue:** `createSceneManager` is constructed in `app.tsx` with `ctx.stores.scene` and `ctx.eventBus`, but internally the module imports `gameStore` and `playerStore` from their singleton module exports. In the real runtime `app.tsx` creates fresh stores via `createGameContext()`, so `ctx.stores.game` and `ctx.stores.player` are **different instances** from the module-level `gameStore` / `playerStore` singletons. This means:
- `loadScene` reads `revealedNpcs` from the wrong store (lines 141, 193).
- `handleInspect` mutates `playerStore` tags on the wrong store (line 271), so item acquisition is not reflected in the context-provided player state.
- The `dialogue_ended` handler (line 112) mutates `gameStore.revealedNpcs` on the singleton, not the context store.

This creates a split-brain state: some state changes go to the module singletons, others to the context stores. Save/restore in `serializer.ts` uses the context stores, so anything written to singletons is lost on save.

**Fix:** Thread `player` and `game` stores through `SceneManager`'s `stores` parameter and use them instead of the imports:

```typescript
// createSceneManager signature
stores: {
  scene: Store<SceneState>;
  game: Store<GameState>;       // add
  player: Store<PlayerState>;   // add
  eventBus?: EventBus;
}
```

### WR-02: Free-text dialogue path does not fire `dialogue_ended` and skips quest advancement

**File:** `src/ui/screens/game-screen.tsx:250-254`

**Issue:** `handleDialogueFreeText` calls `dialogueManager.processPlayerFreeText(text)` directly. This method returns a `DialogueResult | null` but the result is never acted upon — there is no call to `endDialogue()` when the player chooses to end the conversation via free text. More critically, `processPlayerFreeText` does not end the dialogue even after submission; the `dialogue_ended` event only fires when `endDialogue()` is called (either via the menu option "结束对话" or Escape). This means a player who exclusively uses free-text input and never hits Escape will never trigger quest auto-acceptance or stage advancement that depends on `dialogue_ended`. This is the path a new player is most likely to take.

**Fix:** This is a design gap more than a code bug — free text is intended to continue dialogue, not end it. However, the hint text "Esc 结束对话" should remain visible when the user is not in free-text mode. Currently `isFreeTextMode` swaps the hint to "Esc 退出输入" which hides the end-conversation affordance. Ensure the Escape key from the normal (non-free-text) mode always calls `endDialogue` and that players understand that free text is a turn, not a conversation-ending action.

No code change needed in `game-screen.tsx`, but `dialogue-panel.tsx:161` hint text when `!isFreeTextMode` should make the end-dialogue affordance explicit:

```typescript
// dialogue-panel.tsx line 161
<Text dimColor>↑↓ 选择    Enter 确认    直接输入 与NPC对话    Esc 结束对话</Text>
```

### WR-03: `addMemory` in `npc-memory-store.ts` performs two separate `setState` calls — second call may read stale state

**File:** `src/state/npc-memory-store.ts:70-106`

**Issue:** `addMemory` calls `store.setState` to append the entry (line 81), then reads back via `store.getState()` at line 87 to check if eviction is needed, then calls `store.setState` again (line 99) to evict. If any subscriber between those two calls also mutates the same NPC record (unlikely in practice but possible during combat with rapid events), the second `setState` would use a snapshot of `recent` computed from an intermediate state. More importantly, both calls individually trigger the `npc_memory_written` event via the store's `onChange` callback — so every `addMemory` call fires the event twice when eviction occurs, resulting in two disk writes for one logical memory addition.

**Fix:** Merge the two `setState` calls into one:

```typescript
store.setState(draft => {
  if (!draft.memories[npcId]) {
    draft.memories[npcId] = { npcId, recentMemories: [], salientMemories: [], archiveSummary: '', lastUpdated: new Date().toISOString(), version: 0 };
  }
  draft.memories[npcId]!.recentMemories = [...draft.memories[npcId]!.recentMemories, entry];
  const record = draft.memories[npcId]!;
  if (record.recentMemories.length > 15) {
    // eviction logic inline
  }
});
```

### WR-04: `quest_side_missing_ore` will never auto-accept — `auto_accept` + `required_npc_id` logic gap

**File:** `world-data/codex/quests.yaml:64-111` / `src/engine/quest-system.ts:174-195`

**Issue:** `quest_side_missing_ore` has both `auto_accept: true` and `required_npc_id: npc_blacksmith`. In the `dialogue_ended` handler, the auto-accept check at line 183 fires `acceptQuest(questId)` when `template.required_npc_id === npcId`. This path is correct and works. However `quest_side_overdue_debt` has `auto_accept: true`, `required_npc_id: npc_merchant_afu`, and also `min_reputation: -20`. The quest-system's `acceptQuest` gate at line 48 only checks `min_reputation` when `required_npc_id` is also set. For `quest_side_overdue_debt`, the player starts with disposition `0` which is `>= -20`, so the gate passes — this is fine. The actual gap is that the `auto_accept` handler at line 182-184 calls `acceptQuest` but the current guard is `if (progress) continue` — if the quest was previously accepted and then failed, re-acceptance is blocked. Failing a quest makes it permanently inaccessible even if the player re-talks to the NPC. This is a design issue that should be an explicit guard rather than a silent skip.

**Fix:** Either remove auto-re-accept silently, or add a check for `failed` status to allow re-acceptance:

```typescript
// quest-system.ts line 180
if (progress && progress.status !== 'failed') continue;
```

### WR-05: `handleDialogueFreeText` result is fully discarded — NPC response never shown in dialogue history panel

**File:** `src/ui/screens/game-screen.tsx:250-254`

**Issue:** `handleDialogueFreeText` calls `void dialogueManager?.processPlayerFreeText(text)` (the return value is discarded with `void`). `processPlayerFreeText` updates `dialogue.dialogueHistory` in the store, so the NPC's response does appear in the `DialoguePanel` via the store subscription. However, any error thrown is silently swallowed — there's no `.catch()` on the promise. If `processPlayerFreeText` rejects (e.g., LLM unavailable), the player sees nothing and the dialogue state stays active with no feedback.

**Fix:**

```typescript
const handleDialogueFreeText = useCallback(
  (text: string) => {
    dialogueManager?.processPlayerFreeText(text).catch((err: unknown) => {
      // surface error to player
      const msg = err instanceof Error ? err.message : String(err);
      sceneStore.setState(draft => {
        draft.narrationLines = [...draft.narrationLines, `[对话错误] ${msg}`];
      });
    });
  },
  [dialogueManager],
);
```

## Info

### IN-01: `use-npc-dialogue.ts` fires a secondary `generateNpcDialogue` call for "all-defaults" metadata — doubles LLM cost silently

**File:** `src/ui/hooks/use-npc-dialogue.ts:82-97`

**Issue:** When streaming completes and `extractNpcMetadata` returns all defaults (neutral, no remember, no sentiment) on text longer than 50 chars, the hook makes a full `generateNpcDialogue` call to get structured metadata. This is a fallback for the streaming path that cannot extract structured fields. The logic is sound but the trigger condition `isAllDefaults && isSubstantive` will fire on nearly every normal NPC utterance in Chinese (Chinese text at 50+ chars with no obvious emotion keywords). This means most NPC turns cost two LLM calls — one streaming, one structured. This is a cost issue rather than a bug, but players may notice latency spikes.

**Fix:** Consider narrowing the fallback condition or accepting `{ sentiment: 'neutral' }` without a second call for short exchanges.

### IN-02: `metadata-extractor.ts` uses single-character CJK patterns — may over-match

**File:** `src/ai/utils/metadata-extractor.ts:9-16`

**Issue:** The EMOTION_PATTERNS match any occurrence of a single CJK character. For example `['angry', /[怒愤恨]/]` will match the character 怒 inside words like 愤怒 (anger) correctly, but also inside 怒放 (blooming) or 愤慨 (indignant — probably fine). The `suspicious` pattern includes 狐 (fox), which appears in place names and animal descriptions. Low risk for gameplay but can cause incorrect `emotionTag` assignments in narration contexts.

### IN-03: `memory-persistence.ts` is never called from `app.tsx` — NPC memories are never persisted to disk

**File:** `src/persistence/memory-persistence.ts:87` / `src/app.tsx` (all lines)

**Issue:** `initMemoryPersistence(memoryDir)` is exported but there is no call to it anywhere in `app.tsx` or any other wiring file. The `npc_memory_written` event fires correctly from `npc-memory-store.ts` whenever a memory is added, but no listener is registered because `initMemoryPersistence` is never invoked. NPC memories survive within a session (they're in the store), and are saved/restored via the serializer through `npcMemorySnapshot`. However, the incremental per-NPC disk files (`/memory/{region}/{npcId}.json`) are never written — the `memory/index.json` is never created. If the memory system was intended to load per-NPC files on startup (as the index structure implies), the round-trip is broken. Currently no code reads these files on boot so it's latent rather than immediately breaking, but should be addressed before the memory loading path is implemented.
