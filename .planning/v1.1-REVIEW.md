## Summary

Reviewed v1.1 wiring across streaming, animation, character creation, and game-screen integration. No hardcoded secrets, no dangerous functions, and no empty catch blocks were found. The critical issues are both integration gaps: the NPC streaming hook (`useNpcDialogue`) is instantiated in `game-screen.tsx` but its `startDialogue` is never called during normal gameplay (the controller and dialogue-manager use a blocking non-streaming path), and `ChapterSummaryPanel` captures keyboard input unconditionally regardless of whether it is the active overlay.

---

## Critical Issues

### CR-01: NPC streaming hook is wired in game-screen but never triggered during gameplay

**File:** `src/ui/screens/game-screen.tsx:100-106`
**Issue:** `useNpcDialogue()` is instantiated and destructures `npcStreamingText`, `isNpcStreaming`, `npcMetadata`, `skipNpcDialogue`, and `resetNpcDialogue`. The controller receives `resetNpcDialogue` and calls it in `handleNpcDialogueComplete`, but `startDialogue` is never passed to or called from the controller. During actual talk actions the flow is: `talk-handler.ts` → `dialogueManager.startDialogue()` → `generateNpcDialogue()` (blocking) → stores dialogue text directly in `dialogueStore`. The streaming hook never fires, so:

- `npcStreamingText` is always `''` during main game NPC dialogue
- `isNpcStreaming` is always `false`
- The composite streaming text on line 398 (`isNpcStreaming ? \`${dialogueState.npcName}：${npcStreamingText}\` : undefined`) always evaluates to `undefined` when NPCs speak
- `handleNpcDialogueComplete` (line 193) depends on `npcMetadata` being set by the hook, but `npcMetadata` is never populated → the effect guard `!hasFiredRef.current` never passes → NPC dialogue completion callback never fires

Players see NPC text appear as a sudden full block (via the dialogue store) with no streaming animation. The `ScenePanel` spinner shows "正在思考..." during the blocking AI call but then snaps to final text with no typewriter effect, which is inconsistent with narration streaming behavior.

**Fix:** Either:
1. Wire `startDialogue` (from `useNpcDialogue`) into the controller deps alongside `startNarration`, and have the talk/dialogue path call `startDialogue` instead of writing directly to store; or
2. Remove the streaming infrastructure from `game-screen.tsx` for NPC dialogue and accept the blocking path as intentional (then also remove dead `npcStreamingText`/`npcMetadata` wiring on lines 101–106 and 193–197 and 398).

### CR-02: `ChapterSummaryPanel` intercepts all keyboard input even when not active

**File:** `src/ui/panels/chapter-summary-panel.tsx:17`
**Issue:** `useInput` is called without an `{ isActive }` option. This means PgUp/PgDn and Escape keystrokes are captured by `ChapterSummaryPanel` regardless of whether the `chapter_summary` overlay is actually shown. All overlay panels are rendered through `PanelRouter`, which keeps all panels in the `panelMap` memo — including `chapter_summary`. If Ink renders all memo entries (even those not returned), the unconditional `useInput` inside `ChapterSummaryPanel` will conflict with every other key handler. In particular, Escape presses will trigger `onClose` silently while the player is in the main game view.

**Fix:**
```typescript
// chapter-summary-panel.tsx — pass isActive prop and thread it to useInput
type ChapterSummaryPanelProps = {
  readonly summaries: readonly string[];
  readonly onClose: () => void;
  readonly isActive?: boolean;   // add this
};

// inside ChapterSummaryPanel:
useInput(useCallback((...) => { ... }, [onClose, canScrollDown]), { isActive: isActive ?? true });
```

Then in `panel-router.tsx` pass `isActive={phase === 'chapter_summary'}`.

### WR-01: `resolveCharacter` can return empty string `raceId` — `buildCharacter` silently uses it as display name

**File:** `src/engine/weight-resolver.ts:242-243`
**Issue:** `resolveCharacter` returns `{ raceId: weights.raceId, ... }`. `createInitialWeights()` sets `raceId: ""`. If a player completes all 4 rounds without any round's `DialogueOptionEffect` providing a `raceId`, the resolved `raceId` is `""`. In `character-creation.ts:86`, `queryById(codexEntries, "")` returns `undefined`, so `race` is `undefined`. The player state then has `race: "" ` (line 124: `race: race?.name ?? selections.raceId` — fallback to the empty string). The player sees `种族：` with no value on the character sheet and status bar. Profession hits the same issue if all profession weights are zero (returns `""` from `resolveByWeight` line 89).

**Fix:** Add a fallback in `narrative-creation-screen.tsx` before calling `buildCharacter`, or add a default codex ID guard in `resolveCharacter`:
```typescript
// In narrative-creation-screen.tsx handleNameSubmitted, after resolveCharacter:
const safeRaceId = resolved.raceId || dialogueConfig.archetypePriority.profession[0] ?? 'race_human';
const safeProfessionId = resolved.professionId || dialogueConfig.archetypePriority.profession[0] ?? 'profession_wanderer';
```

### WR-02: `skipToEnd` in `use-streaming-text` fires `setStreamingText` twice — once immediately and once from `finally`

**File:** `src/ui/hooks/use-streaming-text.ts:115-122` and `105-109`
**Issue:** When `skipToEnd` is called mid-stream, it sets `skippedRef.current = true` and calls `setStreamingText(fullTextRef.current)` immediately (line 120). The async loop continues running, finishes the stream (chunks are silently discarded because `skippedRef` is true), and then hits the `finally` block (line 105) which calls `setStreamingText(fullTextRef.current)` again (line 108) and `setIsStreaming(false)`. This causes a double state update: a premature render with partial text followed immediately by a second render with the full final text. During in-flight skip, the displayed text may momentarily show incomplete content, then flash to the final complete text.

The double-update is benign in most cases but causes a visible flash on skip — inconsistent with the intended "snap to complete text" UX.

**Fix:** In the `finally` block, gate the second `setStreamingText` call behind `!skippedRef.current`:
```typescript
if (!cancelledRef.current) {
  bufferRef.current?.flush();
  bufferRef.current?.dispose();
  if (!skippedRef.current) {
    setStreamingText(fullTextRef.current);
  }
  setIsStreaming(false);
}
```

### WR-03: `chapterSummaries` passed to `PanelRouter` is a non-reactive read

**File:** `src/ui/screens/game-screen.tsx:395`
**Issue:** `chapterSummaries={getRecentChapterSummaries()}` is called inline during render, reading the module-level `recentChapterSummaries` array from `summarizer-worker.ts`. This is not reactive — if a chapter summary is generated (via `runSummarizerLoop`) while the panel is already open, the `ChapterSummaryPanel` will not re-render to show it. Also, `getRecentChapterSummaries()` is called on every render of `GameScreen`, which is fine for read cost but the returned array identity is the same module-level array reference, so React's shallow equality check will never trigger a re-render even when items are pushed (since the reference doesn't change).

**Fix:** Subscribe to the `summarizer_task_completed` event and maintain React state for chapter summaries, similar to how `lastTurnTokens` is tracked:
```typescript
const [chapterSummaries, setChapterSummaries] = useState(() => getRecentChapterSummaries());
useEffect(() => {
  const handler = (p: DomainEvents['summarizer_task_completed']) => {
    if (p.type === 'chapter_summary') {
      setChapterSummaries(getRecentChapterSummaries().slice());
    }
  };
  eventBus.on('summarizer_task_completed', handler);
  return () => eventBus.off('summarizer_task_completed', handler);
}, []);
```

### IN-03: `replayEntries` memo dependency on `gameState.phase` is a rough proxy

**File:** `src/ui/screens/game-screen.tsx:225`
**Issue:** `const replayEntries = useMemo(() => getLastReplayEntries(), [gameState.phase])` — replay entries only update when the phase changes. If a player opens the replay panel (`'replay'` phase), then closes it, then plays a turn, then re-opens the replay panel in the same phase transition cycle, the memo will return stale entries (the turn just played is not included). The dependency should be `gameState.turnCount`.

**Fix:**
```typescript
const replayEntries = useMemo(() => getLastReplayEntries(), [gameState.turnCount]);
```
