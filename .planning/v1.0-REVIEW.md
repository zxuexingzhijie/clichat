# v1.0 Full Codebase Review

**Reviewed:** 2026-04-29
**Depth:** quick (with targeted content reads)
**Files Reviewed:** 55
**Status:** issues_found

## Summary

Reviewed the full v1.0 Chronicle CLI codebase. The core architecture is sound — immutable state stores, event bus wiring, and the Rules Engine / AI separation boundary are all correct. The most critical issues are: (1) a broken save/load wiring that means `loadGame` never fires `quest_stage_advanced` / `quest_completed` event-bus events after restore, silently corrupting quest auto-advance; (2) the `turn_log_compress` summarizer task dispatches an empty array instead of actual log entries; (3) the `alibaba` provider factory uses `openai.call()` which is not a valid AI SDK v5 API and will throw at runtime; (4) the `questSystem` memo in `app.tsx` depends on `allCodexEntries` which starts as an empty Map and is replaced after async load, meaning the quest system created on initial render has no codex data and will miss quest auto-accept triggers.

Several warnings cover functional gaps that affect gameplay: combat `cast` falls through to attack resolution when no spell heals but the MP was already deducted; the `turn_log_compress` worker always passes `[]`; `save-migrator` `migrateV3ToV4` is a no-op leaving no meaningful upgrade path; and several UI panels have usability gaps. Info items cover minor dead code and labeling inconsistencies.

## Critical Issues

### CR-01: Alibaba provider factory uses invalid AI SDK v5 API

**File:** `src/ai/providers.ts:47`
**Issue:** `openai.call({ ... }, id)` is not a valid method on the `openai` provider object in AI SDK v5. This will throw `TypeError: openai.call is not a function` at runtime whenever any role is configured with the `alibaba` provider.
**Fix:**
```typescript
import { createOpenAI } from '@ai-sdk/openai';
const alibaba = createOpenAI({
  baseURL: process.env.ALIBABA_BASE_URL ?? 'https://dashscope.aliyuncs.com/compatible-mode/v1',
  apiKey: process.env.ALIBABA_API_KEY ?? '',
});
// factory:
alibaba: (id) => alibaba(id) as unknown as LanguageModel,
```

### CR-02: questSystem created before codex loads — misses all auto-accept triggers

**File:** `src/app.tsx:138-145`
**Issue:** `questSystem` is created via `useMemo` that depends on `allCodexEntries`. On first render `allCodexEntries` is an empty `Map` (codex loads asynchronously). When codex finishes loading, `allCodexEntries` changes, which causes a new `questSystem` to be created. However, the new `questSystem` wires fresh `bus.on(...)` listeners but the old one's listeners are never cleaned up — so both old (empty-codex) and new (loaded-codex) instances respond to events simultaneously. The old one can never match any quest template, the new one can. This is a listener leak that doubles event processing indefinitely and may cause duplicate quest-starts.
**Fix:** The `questSystem` should be created once (after codex is ready) and should expose a cleanup function, or codex should be loaded before React renders. At minimum, guard the `useMemo` to return a no-op system when `allCodexEntries.size === 0`:
```typescript
const questSystem = useMemo(
  () => {
    if (allCodexEntries.size === 0) return createNoopQuestSystem();
    return createQuestSystem(...);
  },
  [ctx, allCodexEntries],
);
```
And ensure the old `bus.on` subscriptions are removed when the memo recalculates. Since `createQuestSystem` registers `bus.on` in its body without returning an unsubscribe, there is currently no cleanup path.

### CR-03: `turn_log_compress` dispatches empty array — summarizer always compresses nothing

**File:** `src/ai/summarizer/summarizer-worker.ts:69-72`
**Issue:** The `turn_log_compress` task branch calls `generateTurnLogCompress([])` — hardcoded empty array. The task schema carries `entryIds` but the worker ignores it and never fetches the actual turn log entries. The summarizer LLM receives no content and returns an empty or nonsensical summary that gets stored in `recentTurnCompressBlocks`.
**Fix:**
```typescript
if (task.type === 'turn_log_compress') {
  const { getTurnLog } = await import('../../engine/turn-log');
  const allEntries = getTurnLog();
  const entries = task.entryIds.length > 0
    ? allEntries.filter(e => task.entryIds.includes(String(e.turnNumber)))
    : allEntries.slice(-20);
  const result = await generateTurnLogCompress(entries);
  recentTurnCompressBlocks.push(result);
  return;
}
```

### CR-04: Combat `cast` with non-heal spell deducts MP then falls through to attack damage path

**File:** `src/engine/combat-loop.ts:194-226` and `228-296`
**Issue:** When a non-heal spell is cast (e.g., `effect_type !== 'heal'`), the MP is deducted at line 209, then the code falls through past the `cast` block without returning. It then re-enters the `attack/cast` damage calculation at line 273 using the spell's `base_value` as weapon base — which is the intended behavior. However, if `actionType === 'cast'` and `spell` is `null` (no `spellId` provided, or the codex entry is missing), MP equal to `CAST_MP_COST` is still deducted at line 209, and the fall-through proceeds with `CAST_WEAPON_BASE` for damage. The spell check at line 199 only covers the "unknown spell" guard for `spellId` present but not found. If `spellId` is `undefined` (player types `:cast` with no argument), the code deducts MP and does generic magic damage with zero LLM flavor context — the player loses MP with no feedback about what spell was cast.
**Fix:** Add an explicit guard before MP deduction:
```typescript
if (actionType === 'cast' && !options?.spellId) {
  stores.combat.setState(draft => { draft.phase = 'player_turn'; });
  return { status: 'error', message: '请指定法术名称。' };
}
```

## Warnings

### WR-01: `quest_stage_advanced` / `quest_completed` bus events not emitted on save restore

**File:** `src/state/serializer.ts:164-175`
**Issue:** `restore()` calls `stores.quest.setState(draft => Object.assign(draft, data.quest))` which replaces quest state in bulk. The `questStore` `onChange` handler emits `quest_stage_advanced` and `quest_completed` based on diff between `oldState` and `newState`. During restore all quests transition from their empty-default state to the loaded state simultaneously. This fires spurious `quest_started` events for every active quest on load, and the `questSystem`'s `pendingConditions` Map is reset (it is in-memory only), so any quest stage that requires two events to advance will lose its partial-completion state across save/load cycles.
**Fix:** The serializer should set `isRestoring = true` on the quest store (as `relationStore` does) to suppress event emission during bulk restore. Alternatively, restore quest state without triggering `onChange`.

### WR-02: `sceneManager` and other systems created before codex loads — stale closure over empty Map

**File:** `src/app.tsx:147-163`
**Issue:** `sceneManager`, `dialogueManager`, and `combatLoop` are all created via `useMemo` with `[ctx, allCodexEntries]` as deps. When codex loads and `allCodexEntries` changes, new instances of all three are created and passed to a new `gameLoop`. However, the `questSystem` wiring (CR-02) means the old quest system's event listeners remain active. More concretely: if an AI narration is in flight when codex finishes loading, the `sceneManager` reference in the controller may be stale for one render cycle.

This is a structural issue — the entire dependency tree of heavyweight managers is re-created every time codex data changes. In practice codex only loads once, but the re-creation still means any ongoing async operations in the old instances are orphaned.

**Fix:** Load codex before rendering `AppInner`, or use a `ref` pattern for managers to avoid re-creation.

### WR-03: `quest-system.ts` auto-accept logic has inverted guard for already-active quests

**File:** `src/engine/quest-system.ts:178-181`
**Issue:** The `dialogue_ended` listener checks:
```typescript
if (progress && progress.status !== 'failed') continue;
```
This means: if a quest already exists in any state other than `failed` (i.e., `unknown`, `active`, or `completed`), skip it. The intent is correct for `active` and `completed`, but `unknown` quests are quests that exist in state but were never started. Since `getDefaultQuestState()` initializes with `quests: {}`, the only way a quest has `status: 'unknown'` is if it was explicitly set. Still, if a future migration adds quests with `unknown` status to existing saves, they would be silently skipped during auto-accept.

More importantly: the `auto_accept` path should also skip `completed` quests (it does, since `completed !== 'failed'`), but a `failed` quest being re-accepted on every `dialogue_ended` event is undesirable — once a quest is failed it should stay failed unless explicitly reset.

**Fix:** Change the guard to:
```typescript
if (progress && (progress.status === 'active' || progress.status === 'completed')) continue;
```

### WR-04: `save-migrator` `migrateV3ToV4` is a no-op — v4 schema never validated

**File:** `src/persistence/save-migrator.ts:36-44`
**Issue:** `migrateV3ToV4` only changes `version: 3` to `version: 4`. The serializer writes `SaveDataV4` but `SaveDataV4Schema` is defined as `SaveDataV3Schema.extend({ version: z.literal(4) })` — structurally identical to V3. This means there is no actual schema evolution between v3 and v4. When `migrateV2ToV3` runs on a v2 save, it produces a v3 object which then passes `migrateV3ToV4` and becomes v4 — but v3 saves on disk will have `version: 3`, which `SaveDataV4Schema.safeParse` will reject (wrong literal). The serializer's `restore()` only accepts V4; V3 saves written before the schema change cannot be loaded.

**Fix:** Either keep V3 as the live schema (revert V4 schema to `z.literal(3)`) or ensure `migrateV3ToV4` actually adds fields that differ between versions. For now, `SaveDataV4Schema` should accept both literals, or `restore()` should attempt V3 parse as fallback.

### WR-05: Combat `processEnemyTurn` ignores `checkCombatEnd` result — can leave phase stuck at `check_end`

**File:** `src/engine/combat-loop.ts:501-508`
**Issue:** After `processEnemyTurn` calls `checkCombatEnd()`, if `!endResult.ended`, it advances `roundNumber` and sets phase to `player_turn`. But the caller in `game-screen-controller` (not in scope) may call `processEnemyTurn` without awaiting, or the combat UI may not trigger `processEnemyTurn` at all after a player action that doesn't end combat (e.g., `guard`). The player `guard` path in `processPlayerAction` sets phase to `enemy_turn` but does not call `processEnemyTurn` — that must be triggered externally. If the controller doesn't call it after `guard`, combat is stuck in `enemy_turn` phase.

**Fix:** Verify the controller calls `processEnemyTurn` after every player action that results in `phase === 'enemy_turn'`. A self-contained guard would be to have `processPlayerAction` automatically call `processEnemyTurn` when it sets phase to `enemy_turn`, rather than relying on external orchestration.

### WR-06: `listSaves` reads `branches.json` as a save file and silently skips it

**File:** `src/persistence/save-file-manager.ts:68-89`
**Issue:** `listSaves()` filters all `.json` files in the save directory. `branches.json` is also stored in the same save directory (written by `saveBranchRegistry`). It has no `meta` field, so the `if (meta)` check will silently skip it — this is correct behavior but means `branches.json` is always read and parsed unnecessarily on every `listSaves()` call. More importantly, if `quickSave` and branch registry both live in the same dir, and the player has many saves, this is fine. However, `loadLastSave` in `game-loop.ts:148` calls `listFn(saveDir)` and takes `saves[0]` — the most recent save by timestamp. Since `branches.json` is skipped (no meta), this is safe.

The actual warning: `listSaves` catches all JSON parse errors silently but also catches I/O errors with the same `catch {}`. A disk error on a valid save file will silently drop that save from the list, giving the player a false picture of available saves.
**Fix:** Log a warning (to stderr / error panel) when a `.json` file fails to parse, distinguishing `branches.json` by naming convention from unexpected failures.

### WR-07: `NpcMemoryRecord` eviction in `dialogue-manager.ts` bypasses store `addMemory` — no LRU enforcement

**File:** `src/engine/dialogue-manager.ts:392-420`
**Issue:** `writeMemory()` in `dialogue-manager` directly mutates the `npcMemory` store via `setState` with manual array spreading, reimplementing its own append logic. The `addMemory` function exported from `npc-memory-store.ts` (line 65) already handles LRU eviction correctly. `writeMemory` does NOT call `addMemory`; instead it appends with no length cap:
```typescript
existing.recentMemories = [...existing.recentMemories, newEntry];
```
This means dialogue-written memories bypass the 15-entry cap. The cap in `NpcMemoryRecordSchema` (`z.array(...).max(15)`) will reject the state at save time, causing a schema validation error when saving a game after long dialogue sessions with an NPC.
**Fix:** Replace the `writeMemory` implementation with a call to `addMemory(stores.npcMemory, npcId, newEntry)`.

### WR-08: `combat_ended` event emitted with stale `enemyIds` from `combat-store` `onChange`

**File:** `src/state/combat-store.ts:52-53`
**Issue:** The `combat_ended` event is emitted inside the store's `onChange` callback:
```typescript
bus.emit('combat_ended', { outcome: newState.outcome ?? 'victory', enemyIds: newState.enemies.map(e => e.id) });
```
`newState.enemies` contains all enemies including already-dead ones from previous rounds. The `quest-system` listener for `combat_ended` checks `if (targetId && enemyIds && !enemyIds.includes(targetId))` — since dead enemies remain in the array, this check always includes all original combatants regardless of whether they were "defeated in this fight" or were already dead before it started. This is correct for single-combat encounters but could cause unintended quest advancement in multi-phase encounters where the same enemy ID appears twice.

More critically: `outcome` is read from `newState.outcome` which is set to `null` before combat and only set during `checkCombatEnd`. The `onChange` fires when `active` flips from `true` to `false`. This only happens inside `checkCombatEnd` where `outcome` is set in the same `setState` call — so the outcome should be available. This is likely safe but tightly coupled.

**Fix:** Low-risk change: filter `enemyIds` to only those with `hp <= 0` to more precisely represent "defeated enemies":
```typescript
enemyIds: newState.enemies.filter(e => e.hp <= 0).map(e => e.id)
```

### WR-09: `comparePanel` initial state ignores `compareSpec` when component already mounted

**File:** `src/ui/panels/compare-panel.tsx:168`
**Issue:** `useState<CompareState>(() => buildInitialState(compareSpec))` captures `compareSpec` at mount time. The `useEffect` at line 222-226 also calls `runCompare` when `compareSpec` changes. However, `useState` initializer only runs once. If `compareSpec` changes after mount (e.g., user issues `/compare main..rescue` while the compare panel is already open), `buildInitialState` is not called again — only the `useEffect` fires. The `useEffect` correctly calls `runCompare` so functionally this works, but the initial `state` still starts as `selecting` rather than `loading` for the first render of a spec-driven compare, creating a single-frame flash of the selector UI before the effect fires.

This is a minor visual glitch, not a data corruption issue.

**Fix:** This is acceptable, but can be fixed by initializing state to `{ stage: 'loading' }` when `compareSpec` is defined, which `buildInitialState` already does. The issue is only a one-render flash.

### WR-10: `knowledge-tracker.ts` adds NPC codex entry as knowledge on dialogue end with generic description

**File:** `src/engine/knowledge-tracker.ts:68-76`
**Issue:** `onDialogueEnded` always adds a `heard` knowledge entry for `codexEntryId: npcId`. NPC IDs (e.g., `npc_guard_01`) are valid codex entry IDs, so this correctly links to the NPC codex entry. However, the description is hardcoded as `与 ${npcId} 的对话` using the raw NPC ID rather than the NPC's name. In the codex panel, this renders as "与 npc_guard_01 的对话" which breaks immersion and is technically incorrect since the NPC name is available from the codex. Additionally, every dialogue session creates or upgrades a knowledge entry, meaning even one-word exchanges permanently add the NPC to the player's knowledge log at `heard` status.

**Fix:** Look up the NPC entry from the codex to get the actual name, or at minimum use a human-readable description: `与某人的对话`. The codex is available via the event bus or store — add a `codexEntries` parameter to `initKnowledgeTracker`.

### WR-11: `dialogue-panel.tsx` `isFreeTextMode` state is set from `TextInput`'s `onChange` but `useInput` is always registered

**File:** `src/ui/panels/dialogue-panel.tsx:144-159`
**Issue:** `useInput(handleInput, { isActive: isActive && !isFreeTextMode })` correctly disables the option-list keyboard handler when in free text mode. However, `onChange` on the `TextInput` component fires for every character, checking `value.length > 0` to set `isFreeTextMode`. If the player presses a number key (1-9) while the `TextInput` has focus but before any characters have been typed (empty value), `isFreeTextMode` remains `false`, so `useInput` is active, and the number-key branch (`parseInt(input, 10)`) fires — triggering a dialogue option selection simultaneously with the character being typed into the input. This creates a race condition where the first character typed can both be sent to the free-text input AND interpreted as a shortcut number.

**Fix:** `isFreeTextMode` should be set to `true` as soon as the `TextInput` receives focus, not based on its content length. Use `onFocus`/`onBlur` if `@inkjs/ui TextInput` supports it, or pass `isActive={false}` to `useInput` whenever the TextInput is focused.

### IN-03: `sceneStore` default state has `sceneId: 'placeholder_scene'` — exploration-tracker fires on first render

**File:** `src/state/scene-store.ts:26-43`
**Issue:** The default `sceneId` is `'placeholder_scene'`. When the player starts a new game, `sceneManager.loadScene(DEFAULT_START_LOCATION)` is called and fires a `scene_changed` event. However, if the `exploration-tracker` is initialized before `loadScene` is called, the default scene ID `'placeholder_scene'` is already "current" — no `scene_changed` fires for it. This is fine in practice. However, if a save is loaded with a non-placeholder scene ID, the `state_restored` event fires (in `save-file-manager`) which triggers `sceneManager.loadScene(sceneId)` (in `game-loop.ts:151`). This fires `scene_changed` which marks the restored scene as `visited`. This is correct but means every load re-marks the current location as visited even if it already was.
**Fix:** Low impact — the exploration-tracker guards against downgrading (`if (currentRank >= visitedRank) return`), so re-marking `visited` as `visited` is a no-op. No fix needed.

### IN-04: `command-registry.ts` help text in `game-loop.ts` uses `/` prefix but parser strips `/` prefix

**File:** `src/game-loop.ts:39-58` and `src/input/command-parser.ts:27`
**Issue:** The `HELP_COMMANDS` array shows commands with `/look`, `/go`, etc. The `command-parser.ts` strips the leading `/` before parsing. Both conventions work (`/look` and `look` both parse), but the help text implies `/` is mandatory while `input-router.ts` only routes to the command parser when input starts with `/`. If a player tries `look` as free text, it goes through the intent classifier, not the command parser, potentially returning a different result.

This is a documentation inconsistency, not a functional bug, since the help text is accurate about required syntax.
**Fix:** No functional fix needed. Optionally clarify in help text: "Commands require the `/` prefix."
