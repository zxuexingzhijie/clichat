This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

AI-driven CLI interactive novel game — a text-based RPG where the player interacts through a terminal interface. Not a chatbot; a **world simulation system** where AI handles narration and NPC behavior while product rules control truth, pacing, and playability.

Core loop: **Perceive Scene → Express Intent → System Adjudicates → AI Narrates → World State Updates → Next Turn**

## Architecture

### Layer Model

The system is organized into 7 functional layers, each with distinct responsibilities:

| Layer | Responsibility | AI Involvement |
|---|---|---|
| CLI Interaction | Input/output, layout, keybindings | None |
| Input Understanding | Parse commands + NL intent recognition | Light (intent classification) |
| Rules Engine | Adjudicate outcomes, resource changes, relationship shifts | None — deterministic |
| Narrative Director | Generate narration, atmosphere, suggested actions | Heavy (LLM generation) |
| NPC Mind | Character dialogue, emotions, memory-informed behavior | Heavy (LLM generation) |
| World Codex | Lore, locations, factions, spells, relationships | Retrieval target (RAG) |
| Long-term Memory | Session summaries, NPC episodic memory, encounter logs | Retrieval target + LLM summarization |

**Critical boundary**: AI writes prose and NPC dialogue. AI does NOT decide whether events succeed, resources are consumed, or relationships change — the Rules Engine owns those decisions.

### Dual-Input System

Two parallel input paths coexist:
- **Command channel**: `:look`, `:go north`, `:talk NPC`, `:attack`, `:save`, `:branch` — structured, predictable, efficient
- **Natural language channel**: free-form player expression → intent extraction first, then narration

NL input must go through intent recognition before reaching the narrative layer. Never pass raw NL directly to the LLM for adjudication.

### Three Turn Granularities

- **Scene turns**: observation, dialogue, investigation, trade, stealth
- **Conflict turns**: combat, chase, negotiation deadlock, countdown events
- **Journey turns**: cross-region travel, camp, resupply, time advance, random encounters

### Save/Branch System

Modeled after git: quick save, named save, branch save. Players manage story forks like code branches with `:save`, `:branch`, `:compare`, `:replay`.

## RAG Strategy (Claude Code Skills-Integrated Pattern)

### Why Not Long Context

Long context is an expensive buffer, not a world management solution. Do NOT stuff entire lore documents into system prompts. Use structured retrieval instead.

### Four-Layer Memory Architecture

Following the RAG + tool-based retrieval pattern from Claude Code's architecture:

1. **World Facts Layer** (static, high-stability)
   - Races, professions, factions, locations, history, spells, items
   - Stored as structured documents (the "World Codex")
   - Retrieved on-demand per turn via a Retrieval Planner skill

2. **Session State Layer** (ephemeral, current turn context)
   - Current scene, active combat, recent N turns
   - Lives in the conversation context window

3. **Episodic Memory Layer** (per-NPC, per-player)
   - "NPC remembers player saved them", "player owes merchant money"
   - Stored as structured memory notes keyed by character ID
   - Retrieved when the relevant NPC or event is active

4. **Semantic Summary Layer** (compressed long-term)
   - Chapter summaries, character arcs, long-term relationship trends
   - Generated by a background Summarizer AI role, not the main narrative model

### Skills-Based RAG Integration

Adopt Claude Code's pattern where skills orchestrate retrieval workflows:

- **Retrieval Planner Skill**: Given the current scene, player intent, active NPCs, and active quests, determines which codex entries, NPC memories, and quest states to fetch. Outputs a structured retrieval plan — NOT a prompt dump.
- **Narrative Skill**: Receives adjudicated results + retrieved context, generates 80–180 char Chinese narration. Bound by template: cannot invent world facts, cannot override game state.
- **NPC Skill**: Per-character prompt template. Inputs: identity, goals, known memories, current scene, player action. Outputs: dialogue, emotion tag, memory flag, relationship delta suggestion.
- **Quest Generation Skill**: Region-aware side quest creation with constraints (no main-line spoilers, no lore contradictions).
- **Summarizer Skill**: Background compression of long sessions into chapter summaries and NPC memory notes.

Each skill declares its allowed tools (e.g., Narrative Skill has read-only codex access; Rules Engine has no LLM access). Skills grant scoped retrieval permissions, preventing context leakage.

### Conditional Skill Activation

Following Claude Code's `paths`-based conditional skill pattern:
- Combat skills activate only during conflict turns
- NPC skills activate only when the relevant character is in-scene
- Region-specific flavor skills activate based on current location

### Truth vs. Cognition Separation

Three distinct layers of knowledge that must never be conflated:
- **World Truth**: "General X betrayed the kingdom" (codex fact)
- **NPC Belief**: "Bartender believes General X was framed" (NPC memory)
- **Player Knowledge**: "Player heard General X may have a hidden motive" (player journal)

The RAG system must tag retrieved content with its epistemic level. NPCs retrieve only from their own memory + world truth visible to them. Players see only what they've discovered.

## Multi-Model Strategy

### AI Roles → Model Tiers

| AI Role | Latency Requirement | Model Tier |
|---|---|---|
| Narrative Director | Real-time | Fast balanced model (e.g., GPT-4.1 mini, Gemini Flash, Qwen-Plus) |
| NPC Actor | Real-time | Fast balanced model |
| Retrieval Planner | Real-time | Fast cheap model |
| Summarizer | Background | Any cost-efficient model |
| Quest/Plot Planner | Background | High-quality reasoning model |
| Safety Filter | Real-time | Fast cheap model |

### Cost Optimization

- Static content (world rules, output format, character skeletons, narrative style) → cached/prefixed
- Dynamic content (player input, current scene, retrieval hits) → per-turn tokens
- Summarize aggressively between chapters; never accumulate raw conversation history beyond current session

## World Content Organization

Four-tier content asset model:

| Tier | Stability | Examples |
|---|---|---|
| Core Settings | Constitutional | Races, professions, magic system, political structure |
| Regional Flavor | Semi-stable | Local customs, dialects, materials, weather, hostilities |
| Dynamic Plot | Evolving | Quest chains, NPC secrets, relationship changes, player impact |
| Operational/Seasonal | Temporary | Festivals, limited campaigns, community content |

Content packs: **World Pack** (geography + history), **Rules Pack** (resource systems, profession mechanics), **Story Pack** (quests, NPCs, location events), **Event Pack** (seasonal, community co-creation).

Lore entries carry a "canon tier" tag: Established Canon / Regional Common Knowledge / Scholarly Dispute / Street Rumor. This allows the world to expand without frequent retcons.

## CLI UX Principles

- Layout: Scene panel + Status bar + Suggested actions + Input area
- ASCII/Unicode dual-mode maps
- Keyboard: Tab completion, arrow history, Ctrl-R search, Alt-1..9 quick actions
- Accessibility: high contrast mode, colorblind safe, screen-reader-friendly descriptions
- Auto-prompt save/branch after key plot nodes

## MVP Scope

**Include**: Character creation, scene exploration, dialogue, basic combat, quest journal, world codex, NPC long-term memory, ASCII map, save/branch, story recap.

**Exclude**: Multiplayer persistent world, complex economy, custom magic languages, deep multimodal input, creator marketplace, full mod platform.

## Security Boundaries

- All player input, community content, mod text, and retrieved knowledge chunks are **untrusted input**
- Prompt injection defense: NPC dialogue and narration skills cannot modify game state; only the Rules Engine can
- System prompts and admin lore entries must have tiered visibility — never exposed to player-facing output
- Export/share features must support redaction before publishing

## Project

**Chronicle**

Chronicle is an AI-driven CLI interactive novel game — a text RPG where AI handles narration and NPC behavior while a deterministic Rules Engine controls truth, state, and game pacing. Chinese-first, CLI-first, single-player. Players interact through dual-input (structured commands + natural language), manage story forks like git branches, and explore a persistent world with long-memory NPCs.

**Core Value:** The player must feel they are in a **persistent, consistent world that remembers them** — not a chatbot that reinvents the universe every turn.

### Constraints

- **Tech stack**: TypeScript + Bun runtime (matching Claude Code reference architecture)
- **Terminal UI**: React + Ink for rich terminal rendering
- **LLM**: Multi-provider abstraction — fast models (GPT-4o mini, Gemini Flash, Qwen-Plus) for online path, quality models for background tasks
- **No vector DB**: RAG uses file-based keyword/tag search on YAML codex + JSON memory, not embeddings
- **World data**: Human-readable YAML/JSON, git-diffable, version-controllable
- **CLI parsing**: Commander.js for command routing
- **First world**: Classic Fantasy (medieval, races, magic, factions) — one region with sufficient depth

## Alternatives Considered
| Category | Recommended | Alternative | Why Not |
|----------|-------------|-------------|---------|
| Runtime | Bun | Node.js 22+ | Slower startup, no native YAML, needs tsx/ts-node for TS execution. |
| Runtime | Bun | Deno | Ink ecosystem not tested on Deno. Bun has proven Ink compatibility (Claude Code ships on it). |
| Terminal UI | Ink 7 | blessed / blessed-contrib | Abandoned (no updates since 2020). No React model. No TypeScript support. |
| Terminal UI | Ink 7 | terminal-kit (v3.1.2) | Lower-level imperative API. Would require reimplementing layout engine. No component model. |
| LLM SDK | AI SDK v5 | LangChain.js | Over-engineered for this use case. AI SDK is lighter, better typed, provider system is cleaner. No vector DB means zero LangChain retrieval benefit. |
| LLM SDK | AI SDK v5 | Direct HTTP per provider | Would need to reimplement streaming, structured output, retry logic, error normalization for 5+ providers. AI SDK solves all of this. |
| LLM SDK | AI SDK v5 | AI SDK v6 | API churn (`generateObject` deprecated), primarily benefits web apps with new streaming patterns. V5 stable and actively maintained. |
| Schema | Zod 4 | TypeBox | Zod is AI SDK's required peer dep. Using anything else means maintaining two schema systems in parallel. |
| Schema | Zod 4 | Effect Schema (v0.75.5) | Requires buying into entire Effect ecosystem. Overkill for validation. |
| State | Custom store | Zustand (v5.0.12) | React coupling, unnecessary abstraction for game logic that must be UI-independent and testable. |
| State | Custom store + Immer | Redux Toolkit | Heavyweight boilerplate for a single-player game. No middleware/saga needs. |
| State | Immer (selective) | Immutable.js | Immer works with plain JS objects. Immutable.js requires proprietary data structures that break Zod schema validation and JSON serialization. |
| Event bus | mitt | EventEmitter3 (v5.0.4) | mitt is smaller (200B vs 1.5KB) and has better TypeScript generic support. |
| YAML | yaml (v2.8.3) | js-yaml | `yaml` is actively maintained with full YAML 1.2 spec. js-yaml stuck on YAML 1.1, lower type safety. |
| CLI parse | Commander | yargs | Commander is simpler, lighter, better TS support. Project constraint anyway. |
